# sign-safe v0.4 — Design (combined, phased A→C→B)

> Status: design of record. Research backing: `/Users/rafazaum/sign-safe-post/RESEARCH-v0.4.md`.
> Scope decision (Rafa): **one combined v0.4 spec, phased A→C→B, full scope — implement everything.**

## Goal

Turn sign-safe from a strong-but-offline *library* into a real **end-to-end signing-time gate** that works on modern mainnet transactions, with measured precision, and a clear-signing registry that is the differentiator. Stay inside the hard invariants.

## Hard invariants (NEVER break)

- Core (`skill/src/{decode,roles,classify,outflow,verdict,tlv,banned,squads,classify-inner,digest,registry,reputation}.ts`) stays **pure / offline / deterministic / dependency-free**. Same bytes → same JSON. Never imports `enrich.ts` or network.
- All network/fetch lives ONLY in `enrich.ts` + `cli.ts` (host layer). Injectable fetcher callbacks; tests inject frozen stubs.
- **Fail-closed, escalate-only**: new analysis only ADDS/escalates within a pass. Enrichment never mutates a verdict in place — it produces *better INPUT* for another deterministic offline pass (so a HOLD can clear to SIGN only by re-running over more-complete, verified data; an unknown can only escalate).
- No-network test (`fixtures.test.ts` T11.3) scans core modules for http/fetch — keep new core modules listed.
- After every change: `cd /tmp/ss-v2 && npm install && npm test && npx tsc --noEmit && npm run gen-fixtures` (0 drift).
- Verify EVERY discriminator/program-id vs canonical source before adding. Never guess.

## Phase A — Make enrichment real (P0)

**Problem:** CLI is fully offline and never calls `enrich`. `enrichAlt` is a stub that throws. So Squads clear-signing needs hand-pasted PDA bytes, and v0/ALT txns (most of mainnet) are force-HOLD'd because ALT-sourced roles are `addressVerified=false`.

### A1. ALT account decoder (`enrichAlt`) — bincode
Implement the canonical bincode layout (verified): u32 LE disc (1=LookupTable) · deactivation_slot u64 · last_extended_slot u64 · last_extended_slot_start_index u8 · authority option tag (byte 21: 00/01) · authority 32B (22..54) · padding u16 (54..56) · addresses packed 32B each from **fixed offset 56**; N=(len-56)/32; max 256. Robust rule: read byte 21 tag. Return `{ table, addresses: string[] }`.

### A2. Core channel for resolved ALTs (`VerdictContext.resolvedAltTables`)
New optional `resolvedAltTables?: ReadonlyMap<string, readonly string[]>` (table base58 → ordered resolved addresses). In `roles.ts`, when building `alt:<table>#wN/#rN` synthetic roles, if the table is present in the map and the index is in range, use the real resolved address and set `addressVerified=true` (and run reserved-key demotion / recipient identity now that the address is known). Fail-closed: table absent OR index out of range → role stays synthetic + `addressVerified=false` (→ still HOLD). This is the only way the over-HOLD clears, and only when ALL ALT roles resolve AND nothing else flags.

### A3. Wire Squads enrichment (`enrichSquads` already works)
Extract the VaultTransaction PDA from account index 2 of the `vaultTransactionExecute` instruction (disc `[194,8,161,87,153,164,25,171]`), fetch its bytes, re-run `reviewBase64(b64, ctx, vtBytes)`.

### A4. Token-2022 mint STATE tier (`confirmMintExtensions`)
Implement the Token-2022 mint extension TLV decode (after 82-byte mint + account-type byte): type 3 = PermanentDelegate (32B), type 4 = TransferHook (program id). For mints touched by the tx, fetch mint account, decode extensions, surface PermanentDelegate/TransferHook as a HOLD-class enrichment finding (invisible offline; real $50M+ scam). Feed via a new context channel (e.g. `mintExtensions?: Map<mint, {permanentDelegate?, transferHook?}>`) consumed by classify/verdict, fail-closed (absent = unchanged behavior).

### A5. RPC fetcher + CLI wiring
- `skill/src/rpc.ts` (host layer, impure): a plain `fetch` JSON-RPC `getAccountInfo` (base64) → `AccountFetcher`. No baked-in default RPC (fail-closed: without `--rpc`, behavior is byte-identical to today, fully offline).
- CLI flags: `--rpc <url>` (enables enrichment), `--vault-pda <pubkey>` (override auto-extract). When `--rpc` is set: auto-resolve ALTs (A1→A2), auto-fetch Squads PDA (A3), auto-fetch Token-2022 mints (A4), then run the enriched offline pass. Without `--rpc`: unchanged.
- Integration tests use a frozen in-memory fetcher (real RPC responses captured once, committed) → fully offline + deterministic.

**A success criteria:** a real v0 Jupiter swap with ALTs, given `--rpc`, resolves to SIGN (was HOLD); a real Squads proposal hiding an authority change, given `--rpc`, auto-clear-signs the inner ix to REJECT/HOLD naming it; core+tests stay offline; tsc clean; 0 fixture drift.

## Phase C — Precision: benign + malicious corpus (P1)

**Goal:** measure false-positive (false-REJECT ≈ 0 on benign) and recall (≥95% on AAT/STMT) on frozen, offline, reproducible corpora — and prove the ALT-resolution win.

- **Benign corpus:** `getBlock(slot,{maxSupportedTransactionVersion:0,transactionDetails:"full",encoding:"jsonParsed"})` from a fixed set of historical slots (jsonParsed carries `meta.loadedAddresses` = resolved ALTs) + a curated labeled set (Jupiter v0 swap, Orca LP, Tensor buy, Squads approve, Marinade stake, ATA create, SOL/SPL transfer, Token-2022 benign transfer, owner self-approve). Filter Vote txns. Stratified ~200.
- **Malicious corpus:** built from the documented techniques (SetAuthority→AccountOwner, Assign/AssignWithSeed, Approve→attacker, multi-transfer sweep, durable-nonce-anchored, Token-2022 PermanentDelegate mint, malicious-CPI, Squads-wrapped Assign), patterned on the Lighthouse `blackhat` catalog + incident shapes. ~60.
- **Freeze:** per fixture `{rawTxBase64, loadedAddresses, mintAccounts:{}, label, expectedVerdict}` + slot + manifest checksum. Tests read disk only; no network.
- **Report:** confusion matrix per family (TP/FP/TN/FN, precision, recall); TWO FP buckets — false-REJECT (~0) and over-HOLD (split by "ALT unresolved" vs other); per-family recall (AAT/STMT visible). Deliberate sub-test: same Jupiter swap with ALT resolved vs unresolved to quantify the over-HOLD reduction. Output a committed `docs/precision-report.md`.

**C success criteria:** committed corpus + harness + report; false-REJECT = 0 on benign; AAT/STMT recall ≥95%; ALT-resolved over-HOLD measurably lower than unresolved.

## Phase B — Clear-signing registry expansion (the moat) (P1)

**Goal:** per-instruction danger labels for the top protocols ("ERC-7730/sRFC-39 for Solana", structural fallback).

- Discipline: for each program, fetch the **on-chain Anchor IDL** (`anchor idl fetch`) and/or canonical GitHub IDL/source, DIFF, and derive discriminators — Anchor: `sha256("global:<snake>")[..8]`; native tag-byte for Raydium AMM v4 (u8 enum), Metaplex Token Metadata (Shank), Sanctum (Solores). Verify every byte. A small build script computes/validates discriminators from names so we never hand-type bytes.
- Priority: (1) Jupiter v6 `JUP6…` (2) Pump.fun `6EF8…` + Pump AMM `pAMM…` (3) Raydium CLMM `CAMM…`/CPMM `CPMM…`/AMM v4 `675k…` (4) Drift `dRifty…n33UH` (5) Kamino klend `KLend…`. Then Orca, Meteora DLMM/DAMM, Marginfi, Tensor TCMP, Magic Eden M2, Sanctum.
- Danger labels focus on authority/withdraw/admin/close/delegate instructions (per research §4).
- Fail-closed: recognized program + unrecognized instruction → HOLD (never SIGN). Recognized danger ix → escalate.

**B success criteria:** registry expanded with verified entries (discriminators validated against canonical IDL in a test); known authority-change ix for each protocol decoded + labeled; fail-closed preserved; tests green; 0 drift.
